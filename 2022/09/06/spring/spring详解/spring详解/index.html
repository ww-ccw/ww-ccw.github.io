Spring详解

Bean

Bean是什么

  1. 简介

    在 Spring 中，构成应用程序主干并由Spring IoC容器管理的对象称为bean。bean是一个由Spring IoC容器实例化、组装和管理的对象。bean是一个Java对象，根据bean规范编写出来的类，并由bean容器生成的对象就是一个bean。（不确定，也有说bean和类是一对一关系，是特殊的类）

    在此意义上@Controller、@Service、@Mapper都有类似于@Bean的用处，即告诉spring这个类是控制器、服务层类、持久层类

    bean规范：

    1. 所有属性为private

    2. 提供默认构造方法

    3. 提供getter和setter

    4. 实现serializable接口

    5. 默认是单例的（包括@Controller、@Service、@Mapper、@Component…）

    6. 可以使用@Scope(“prototype”)声明这是一个原型而不是单例

      • singleton：默认属性， 这个bean会被定义为一个单例对象，第一次被注入时，会创建为一个单例对象，该对象会一直被重复使用直到这个功能结束，该对象的生命周期是与Spring IOC容器 。

      • prototype：bean会在每次注入时都会创建一个新的对象

      • request：bean会在每个HTTP请求中创建一个单例对象，也就是说在单个请求中都会复用这一个单例对象。

      • session：bean会在一个session的生命周期内创建一个单例对象

      • application：bean会为在ServletContext的生命周期中复用一个单例对象。

      • websocket：bean会为在websocket的生命周期中复用一个单例对象。
        例如：

        @Component
@Scope("prototype")			//"prototype"非单例
public class User {
    private String user;
    private int age;
}

  2. Bean容器

    或称spring ioc容器，主要用来管理对象和依赖，以及依赖的注入。在 Spring 中，类的实例化、依赖的实例化、依赖的传入都交由 Spring Bean 容器控制，而不是用new方式实例化对象、通过非构造函数方法传入依赖等常规方式。实质的控制权已经交由程序管理，而不是程序员管理，所以也叫控制反转IOC。

Bean的单例是如何实现的

Spring通过单例注册表来实现单例，IOC容器维护一个bean表格、当需要一个单例bena时，从表格中获取、没有获取到的，向表格注册一个新的bean

单例注册表是通过一个ConcurrentHashMap对象实现的。

Bean默认单例带来的线程安全问题

问题：Bean默认单例那该如何解决在多线程环境中实现线程安全问题？

  1. 在Bean中不要使用可能造成线程安全问题的成员变量。
  2. 对于必要的需要的变量尽可能的使用局部变量，这样就可以在方法内产生，在方法内消亡
  3. 再万一有非使用不安全变量的时候可以使用ThreadLocal，这样可以使每个线程有独属于自己的存放空间。

IOC和DI

IoC本质

控制反转IoC(Inversion of Control)，是一种设计思想，DI(依赖注入)是实现IoC的一种方法，原本需要由程序员手动new的对象可以由spring框架为你自动创建并注入，也就是获得对象的方法反转了。IOC容器支持饿汉式和懒加载

在早期spring中配置bean需要在xml文件中书写bean，而在springboot中则不需要去配置xml文件，只须使用注解标注就行。

最大的优点在于解耦，如果要修改代码的时候不需要牵一发而动全身，到处修改。由容器自动维护具体对象

AOP

简介

AOP（Aspect Oriented Programming）意为：面向切面编程，通过预编译方式和运行期动态代理实现程序功能的统一维护的一种技术。AOP是OOP的延续，是软件开发中的一个热点，也是Spring框架中的一个重要内容，是函数式编程的一种衍生范型。利用AOP可以对业务逻辑的各个部分进行隔离，从而使得业务逻辑各部分之间的耦合度降低，提高程序的可重用性，同时提高了开发的效率

执行流程（Spring2.x）

  1. Around 环绕
  2. Before 前置
  3. 业务逻辑
  4. AfterReturning/AfterThrowing 返回后（未到前端）/异常后
  5. After 后置（finially）
  6. Around 环绕

注：Spring1.x的4、5顺序交换

三级缓存

public class DefaultSingletonBeanRegistry extends SimpleAliasRegistry implements SingletonBeanRegistry {
    /** 一级缓存：单例对象的缓存：bean 名称到 bean 实例. */
    private final Map<String, Object> singletonObjects = new ConcurrentHashMap<>(256);

    /** 二级缓存：早期单例对象的缓存：已经实例化但未初始化。bean的属性未完全填充 */
    private final Map<String, Object> earlySingletonObjects = new ConcurrentHashMap<>(16);

    /** 三级缓存：单例工厂的缓存：创建Bean的工厂. */
    private final Map<String, ObjectFactory<?>> singletonFactories = new HashMap<>(16);
}

单例Bean的创建就时依赖于三级缓存

一级缓存：存放所有已经的完全创建好的的Bean，别名单例池

二级缓存：存放已实例化，但没有初始化的Bean，（这也是解决循环依赖的手段，也只有单例的Bean可以通过提前暴露的方式解决循环依赖，而非单例的Bean每次都需要一个新的对象，所以非单例的Bean时没有缓存的）

三级缓存：存放生成单例Bean的工厂

循环依赖

假如有A和B两个类互为循环依赖，那么创建的过程将会是

  1. A创建时发现需要B，A先放在三级缓存中，然后去实例化B
  2. B实例化时发现需要A，那么就先去一级缓存中找，然后去二级缓存中找，最后去三级缓存中找，发现A后，先将A放到二级缓存中，同时删除三级缓存中的A
  3. B顺利初始化后，将自己放在一级缓存中（此时B中的A还在创建中状态）然后回来创建A，此时B已经创建完成，直接从一级缓存中拿到B完成创建，将A放入一级缓存中

AB循环依赖问题只要A的注入方式是setter且singleton ，就不会有循环依赖问题。

SSM流程图

  1. 用户发送请求至 前端控制器DispatcherServlet。
  2. 前端控制器DispatcherServlet收到请求后调用处理器映射器HandlerMapping。
  3. 处理器映射器HandlerMapping根据请求的Url找到具体的处理器，生成处理器对象Handler及处理器拦截器HandlerIntercepter（如果有则生成）一并返回给前端控制器DispatcherServlet。
  4. 前端控制器DispatcherServlet通过处理器适配器HandlerAdapter调用处理器Controller。
  5. 执行处理器（Controller，也叫后端控制器）
  6. 处理器Controller执行完后返回ModelAnView。
  7. 处理器映射器HandlerAdapter将处理器Controller执行返回的结果ModelAndView返回给前端控制器DispatcherServlet。
  8. 前端控制器DispatcherServlet将ModelAnView传给视图解析器ViewResolver。
  9. 视图解析器ViewResolver解析后返回具体的视图View。
  10. 前端控制器DispatcherServlet对视图View进行渲染视图（即：将模型数据填充至视图中）
  11. 前端控制器DispatcherServlet响应用户。
